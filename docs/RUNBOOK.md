# STS Test Framework – Quick start

**Preferred path:** After install, use the **STS Test Runner web UI** — double-click `launcher.command` (macOS) or `launcher.bat` (Windows), or run `python launcher.py` from the project root. Choose QA/stage/prod and the suite to run; reports still land under `reports/`. Full walkthrough: **[ONBOARDING §5.0](ONBOARDING.md#50-web-test-runner-ui-recommended)**.

**This document** is the **CLI / shell script backup** for automation, pytest flags, forwarded arguments, and environment variables. Use it when you are not using the UI.

---

## Web UI (quick)


| Platform    | Start                                                                  |
| ----------- | ---------------------------------------------------------------------- |
| **macOS**   | Double-click `launcher.command`, or `python launcher.py` (venv active) |
| **Windows** | Double-click `launcher.bat`, or `python launcher.py` (venv active)     |


Default URL: **[http://localhost:5678](http://localhost:5678)** (launcher picks the next free port if busy). Same underlying scripts as below — outputs match the command-line workflow.

**Windows:** Full, manual, and term-verify suites invoke `bash`; install **Git Bash** (or ensure `bash` is on `PATH`) for those. **Autogenerated** runs without `bash`.

If you would like to use the command line instead to run tests, continue below.

---

## Prerequisites

From the **project root**:

```bash
pip install -r requirements.txt
pip install -e .
```

`requirements.txt` includes **boto3** for the optional **parser_agent** step: an **AI (Amazon Bedrock) log parser** that reads captured run logs and, when failures are present, writes Markdown summaries under **`reports/agent-summaries/*.md`** (only when **`AWS_ACCESS_KEY_ID`**, **`AWS_SECRET_ACCESS_KEY`**, and **`AWS_REGION`** are all set). Without those variables, the convenience scripts skip the parser. The same boto3 stack is available via `pip install -e ".[agent]"` if you prefer not to use the flat file. If you use **uv**, ensure boto3 is present when you enable the parser.

You need **network access** to STS for API tests and term-verify runs.

## Configuration (optional)

**Default:** If you do **not** set anything, the framework uses **QA**: `https://sts-qa.cancer.gov/v2` (see [sts_base_url()](../src/sts_test_framework/config.py)). You can run the scripts below with no `export`.

**Other environments:** Set **`STS_BASE_URL`** only when you want something other than QA (stage, prod, local dev, etc.). STS v2 root must include `/v2`. The web UI sets this for you per run when you pick a preset.

```bash
export STS_BASE_URL=https://sts.cancer.gov/v2   # example: prod
```

Same effect without `export` for one command: `STS_BASE_URL=https://... bash scripts/run_manual_tests.sh`. After `pip install -e .`, the **`sts-test`** CLI also accepts `--base-url` (details: [ONBOARDING §5.2](ONBOARDING.md#52-configuration-environment-variables)).

When you run **manual** tests via `run_manual_tests.sh`, pytest prints `STS environment: https://...` — use that line to confirm which host you hit.

**More options** (`STS_SSL_VERIFY`, `REPORT_DIR`, `STS_MODELS`, `STS_DEDUP_LIMIT`, etc.): [ONBOARDING.md §5.2](ONBOARDING.md#52-configuration-environment-variables).

## Three scripts (default workflow)

All commands assume the **project root** as the current directory.


| Script                                      | What it runs                                                                                                                                                                      | Main output                                       |
| ------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------- |
| `bash scripts/run_manual_tests.sh`          | Manual pytest under `tests/test_manual/` with **pytest-html**                                                                                                                     | `reports/manual_tests.html`                       |
| `python scripts/run_autogenerated_tests.py` | [sts_test_framework.cli](../src/sts_test_framework/cli.py) **once per data model**, **sequential** by default (1 worker); optional parallel via `STS_PARALLEL_WORKERS` (e.g. `8`) | `reports/<ModelHandle>/report_*.html` and `.json` |
| `bash scripts/run_all_term_verify.sh`       | Every `tests/term_verify/*_term_verify.py` (YAML enums → STS term endpoints); **2 at a time** by default (`STS_TERM_VERIFY_WORKERS=1` for sequential)                             | `reports/term_value/<COMMONS>/` per commons       |


**Extras:**

- **Manual script:** arguments are forwarded to pytest, e.g. `bash scripts/run_manual_tests.sh -m nullcde`.
- **Autogenerated runner:** limit models with `STS_MODELS=PSDC,CTDC python scripts/run_autogenerated_tests.py`. Default is **sequential**; set `STS_PARALLEL_WORKERS=8` (or `2`, etc.) for parallel model runs (see [ONBOARDING §5.6](ONBOARDING.md#56-running-all-data-models-in-one-go-multi-model-runner)).
- **Term-verify:** arguments are forwarded to each script, e.g. `bash scripts/run_all_term_verify.sh --warn-only` or `--limit 50`. Default **2 concurrent** commons pipelines (`STS_TERM_VERIFY_WORKERS`). Per-commons scripts and report paths: [ONBOARDING §3.8](ONBOARDING.md#38-term-by-value-yaml--sts).

## Run all three in one command

```bash
bash scripts/run_full_suite.sh
```

Runs **manual → autogenerated → term-verify** in that order **every time** (later stages still run if an earlier one fails). Prints a **summary** of which stages passed or failed; **exit code 1** if any stage failed. It does **not** forward extra arguments; use each script separately for filters and flags.

## Optional: full pytest

To run **generated + manual + unit** together under pytest (terminal-focused; not the same as the per-model CLI HTML reports), see [ONBOARDING.md §5.4](ONBOARDING.md#54-run-with-pytest-recommended-for-day-to-day-work):

```bash
pytest tests/ -v
```
