# STS v2 API Test Framework — Architecture Review

**Date:** March 2026
**Reviewer perspective:** Senior SDET / Software Architect

---

## Executive Summary

This is a well-structured, thoughtfully documented API test automation framework. The OpenAPI-driven generation pipeline (`loader → discover → generator → runner → reporter`) is a strong foundation. The documentation (especially `ONBOARDING.md`) is unusually thorough for a test framework. However, there is one glaring structural problem that accounts for roughly 3,000+ lines of unnecessary code, a few real bugs, and several design decisions worth revisiting.

## 1. Critical: Term Verify Code Duplication — DONE

> **Resolved:** Extracted `TermVerifyPipeline` base class (`src/sts_test_framework/term_verify_pipeline.py`) and shared utilities (`src/sts_test_framework/term_verify_utils.py`). All 6 files rewritten as thin subclasses. Total code reduced from ~3,489 lines to ~1,416 lines (59% reduction).

Six `*_term_verify.py` files under `tests/term_verify/` contained massive code duplication:

| File | Lines |
|------|-------|
| `ccdi_term_verify.py` | 681 |
| `ccdi_dcc_term_verify.py` | 722 |
| `c3dc_term_verify.py` | 546 |
| `cds_term_verify.py` | 459 |
| `ctdc_term_verify.py` | 533 |
| `icdc_term_verify.py` | 548 |
| **Total** | **~3,489** |

These files are 80-90% identical. The following functions are copy-pasted with only `MODEL_HANDLE` and CSV filenames changed:

- `_repo_root()` — identical in all 6
- `discover_*_version()` — identical except `MODEL_HANDLE`
- `build_prop_to_node_map()` — literally identical in all 6
- `run_enrich()` — nearly identical (CCDI/C3DC/ICDC/CTDC do handle-to-value mapping; CDS skips it; CCDI-DCC adds URL expansion)
- `run_verify()` — nearly identical (minor column differences)
- `main()` — the argparse CLI boilerplate is copy-pasted 6 times

The only real variation between files is:

1. **YAML parsing** — Different regex patterns for `Enum:` headers and how quotes/comments are stripped.
2. **CDS doesn't do handle-to-value enrichment** (uses `enum_value` directly as URL param).
3. **CCDI-DCC fetches remote URLs** during extract and has `KNOWN_MISSING_IN_STS_DB`.
4. **ICDC uses `clean_enum_value`** instead of `strip_inline_yaml_comment`.

Additionally, `verify_row` and `strip_inline_yaml_comment` live in `ccdi_term_verify.py` and are imported by the other files — this creates an odd dependency where CCDI is treated as "canonical."

**Recommendation:** Extract a shared `TermVerifyPipeline` base class. Each model becomes a thin subclass (~30-80 lines) that only defines `parse_yaml` and model-specific overrides. Total: ~600-800 lines instead of ~3,500. Move `verify_row` and `strip_inline_yaml_comment` to a shared utility module in `src/sts_test_framework/`.

## 2. Bug: Dead Code in `_check_basic_shape` — DONE

> **Resolved:** Changed `"nanoid"` to `schema_ref` on line 375 of `functional.py`.

**File:** `src/sts_test_framework/runners/functional.py` line 375

```python
if "nanoid" in ("Entity", "Term", "Node", "PropertyResponse", "Tag") and schema_ref != "Model":
```

The expression `"nanoid" in ("Entity", "Term", "Node", "PropertyResponse", "Tag")` is always `False` — it checks if the string `"nanoid"` is literally one of those schema names, which it never is. The intent was clearly `schema_ref in (...)`. This means the nanoid/value/key shape check never executes and has been silently passing responses that might have invalid shapes.

**Fix:** Change `"nanoid"` to `schema_ref`.

## 3. Bug: Environment Variable Inconsistency — DONE

> **Resolved:** `parser_agent/config.py` now checks `AWS_REGION` first, falls back to `AWS_DEFAULT_REGION`, then defaults to `us-east-1`.

There was a mismatch between how the parser agent checks for AWS credentials:

- `parser_agent/config.py` reads `AWS_DEFAULT_REGION` (line 13)
- `scripts/parser_agent_hook.sh` checks for `AWS_REGION` (line 11)
- `scripts/run_autogenerated_tests.py` checks for `AWS_REGION` (line 128)

If a user sets `AWS_REGION` (as the shell scripts tell them to), the Python code falls back to `"us-east-1"` because `AWS_DEFAULT_REGION` is unset. If a user sets `AWS_DEFAULT_REGION`, the shell gate won't pass because `AWS_REGION` is unset.

**Fix:** `config.py` should check `AWS_REGION` first, fall back to `AWS_DEFAULT_REGION`, then default.

## 4. Design: Test Collection Makes Live API Calls — TODO

`tests/test_generated/test_from_spec.py` runs full API discovery during pytest collection (before any test executes) in `_get_generated_cases()`. If STS is unreachable, tests don't appear as "skipped" or "error" — they simply don't exist, and `pytest --collect-only` makes network calls.

Meanwhile, `tests/conftest.py` has session-scoped fixtures that do the exact same work at execution time. So discovery happens twice when running via pytest.

**Recommendation:** Use fixture-based collection or catch errors and `pytest.skip()` gracefully.

## 5. Design: Untyped Case Dicts — TODO

Throughout the framework, test cases are plain `dict[str, Any]` with ~15 possible keys. There's no `TypedDict`, `dataclass`, or schema definition. A typo in a key name silently produces wrong behavior. New contributors must read the generator code to know what keys exist.

**Recommendation:** Introduce a `TestCase` dataclass or TypedDict.

## 6. Design: Duplicated URL-Building Logic — DONE

> **Resolved:** Extracted `_build_query_string()` in `client.py`. All three call sites (`_make_request`, `full_url`, `_path_with_query`) now delegate to it.

Query string construction appeared in three separate places:

1. `APIClient._make_request()` in `client.py` (lines 61-71)
2. `full_url()` in `client.py` (lines 116-131)
3. `_path_with_query()` in `functional.py` (lines 178-193)

All three do the same `for k, v in params.items()` / `quote(str(v))` / join with `&` logic.

**Fix:** Extract a shared `_build_query_string()` function and have all three call it.

## 7. Design: Dead Code in `_negative_path_params` — DONE

> **Resolved:** Simplified function body to remove the dead `else` branch. All path params now unconditionally receive the invalid value.

**File:** `src/sts_test_framework/generator.py` lines 504-526

The `else` branch (line 521) checks `if name == "id"` — but `"id"` was already caught by the `if name in (...)` check above. Both branches of the inner `if/else` do the same thing (`values[name] = invalid`). The entire `else` block collapses to just `values[name] = invalid`.

## 8. Minor: `run_autogenerated_tests.py` Patches sys.stdout — DONE

> **Resolved:** Removed `_TeeWriter` class and `sys.stdout`/`sys.stderr` patching. Replaced with `_log_and_print()` helper that writes to both console and log buffer without monkey-patching streams.

**File:** `scripts/run_autogenerated_tests.py` lines 184-185

Monkey-patching global I/O streams with a `_TeeWriter` class is fragile. It can interfere with subprocess output, third-party libraries, and pytest's capture system. Since `_run_model` already uses `subprocess.run(capture_output=True)`, all model output is already captured in stdout/stderr strings. Collect output into a buffer directly instead of patching streams.

## 11. Missing: No Retry Logic or Rate Limiting — TODO

The `APIClient` makes a single attempt per request. For an integration test framework hitting a real API, transient 503s or timeouts cause flaky test failures. The term-verify pipelines make thousands of sequential requests with no backoff.

Even a simple retry-on-5xx with exponential backoff (2 attempts, 1-second delay) would significantly improve reliability.

---

## 12. Missing: No Linting/Type Checking Configuration — TODO

There is no `ruff.toml`, `mypy.ini`, `pyright` config, or `.pre-commit-config.yaml`. Adding at minimum `ruff` for linting and `mypy --strict` for the `src/sts_test_framework/` package would catch bugs (like the `"nanoid" in (...)` issue above) before they ship.

## 13. What's Good

1. **The OpenAPI-driven generation pipeline is well-designed.** `loader → discover → generator → runner → reporter` is a clean pipeline with clear separation of concerns.

2. **Documentation is exceptional.** The 844-line `ONBOARDING.md` is comprehensive and well-organized with audience-specific reading paths.

3. **Session-scoped fixtures are correct.** Avoiding redundant API calls during test setup shows good pytest understanding.

4. **Unit tests for runner edge cases.** Testing `_pagination_pair_check` and `_check_basic_shape` with mocked `APIResponse` objects is the right approach.

5. **The parser_agent integration is clever.** Post-run failure analysis via Bedrock is a nice CI convenience. The non-blocking design (always exits 0) is correct.

6. **Shell scripts are well-structured.** The sourced `parser_agent_hook.sh` pattern avoids duplication across scripts. `run_all_term_verify.sh` handles parallelism, failure tracking, and temp file cleanup correctly.

7. **The `--warn-only` flag** on term-verify scripts is practical for refreshing artifacts without blocking CI.

8. **The `--release` flag** for version selection shows attention to real-world model versioning complexity.

## 15. Term Verify: Integration vs Refactor Analysis

The term-verify pipelines were ported from existing test scripts. The question of whether they should be integrated into the manual pytest tests was considered. The recommendation is **do not integrate into pytest** — instead refactor via shared base class:

**Why not integrate into pytest:**

1. They are fundamentally different in kind — data-driven pipelines that generate thousands of rows, not assertion-based API behavior tests.
2. Pytest parametrize would create thousands of test items with unwieldy collection output.
3. The CSV artifact generation (summary, enriched, report) is part of the value — pytest doesn't naturally produce these.
4. The `--skip-extract` / `--skip-enrich` workflow doesn't fit pytest's model.
5. The effort-to-value ratio is poor.

**What to do instead:**

1. Extract a shared `TermVerifyPipeline` base class with the common extract/enrich/verify/main patterns.
2. Each model becomes a thin configuration layer (~30-80 lines each instead of 500-700).
3. Move `verify_row` and `strip_inline_yaml_comment` to `src/sts_test_framework/` (shared utilities).
4. Keep them as standalone CLIs but eliminate the 80% duplication.
