# CCDI term-by-value verification artifacts

Generated by the **CCDI term pipeline** (`python tests/term_verify/ccdi_term_verify.py`), committed as a snapshot for traceability.

| File | Description |
|------|-------------|
| `ccdi_enum_properties_summary.csv` | One row per YAML property with an `Enum` block |
| `ccdi_enum_terms_for_verification.csv` | One row per `(prop_handle, enum_value)` from YAML |
| `ccdi_enum_terms_for_verification_enriched.csv` | Plus STS `model_handle`, `version_string`, `node_handle`, API `term_value` |
| `ccdi_term_endpoint_verification_report.csv` | Per-row HTTP result vs `GET .../term/{termValue}` |
| `ccdi_term_endpoint_verification_report.md` | Summary + failed rows (preview) |

## Regenerate

From repo root (network to STS required; full run ~many minutes):

```bash
pip install -e .
python tests/term_verify/ccdi_term_verify.py
```

Options:

- `--limit N` ‚Äî verify only the first N rows (fast smoke).
- `--skip-extract` / `--skip-enrich` ‚Äî reuse existing CSVs in this folder.
- `--warn-only` ‚Äî exit 0 even if some rows fail (reports still list failures).
- `--yaml PATH` / `--out-dir PATH` ‚Äî override defaults.

Failing rows usually mean: STS has no term for that **value**, or the YAML enum line is not a real term **handle**.

**Pipeline rules (CCDI):**

- Enum lines strip inline `#` YAML comments *outside* quoted strings so values like `"Data Submitter" # note` become handle `Data Submitter`.
- Enrich calls `/terms` with **pagination** (`skip`/`limit`) so large enums resolve every handle‚Üívalue (e.g. `T4` ‚Üí `T4 Stage Finding`).
- Verify uses **only** the enriched `term_value` (API **value** field) in `/term/{termValue}` ‚Äî never `enum_value` (handle).
